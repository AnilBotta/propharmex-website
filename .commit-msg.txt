feat(positioning): remove region/personalization layer (single unified site)

PR-B′ of the positioning brief (2026-05-02). Strips the entire Canada/India/Global region personalization layer per client direction: one unified Propharmex website for all visitors, no geo-detection, no region switcher, no region-keyed content variants.

Why
- Client direction: "single unified Propharmex website for all visitors. Do not create separate versions of content for Canada, India, US, or Global audiences. Do not show region-specific CTAs, region-specific hero copy, or region-specific certification emphasis."
- The personalization layer (Prompt 22) is incompatible with the new specialty-CDMO positioning that PR-C′ will land.

Removed
- Edge middleware: x-vercel-ip-country geo-detection + px-region cookie writes
- React UI: RegionSwitcher (header + footer variants), RegionDetectionBanner, RegionContext/RegionProvider
- API: POST /api/region (cookie writer)
- Server helpers: getServerRegion, shouldShowRegionBanner
- Content variants: home-region.ts (regionalizeHome), quality-region.ts (regionalizeQuality), region.ts (REGION_COPY)
- Library: packages/lib/region/* (Region type, country-map, region-aware helpers, tests)
- Analytics: region super-property in PostHog register(); region-cookie helpers; region-telemetry events (region.detected/changed/banner_dismissed)
- Tests: 3 region-specific test files
- Site-nav: REGIONS export + Region type re-export from @propharmex/lib/region
- Docs: region rows + region middleware section + region breakdown dashboard from analytics-taxonomy.md

Preserved
- "Headquartered in Canada" copy as factual company positioning
- Form-side regulatory-region selector on InquiryForm (different concept: user-input form data)
- CaseRegion filter on /case-studies (case-study metadata)
- addressRegion JSON-LD properties on facility pages (geographic metadata)
